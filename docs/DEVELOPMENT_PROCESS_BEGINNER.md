# Claude Code로 개발하는 법 — 초심자 가이드

> "AI가 코드를 짜준다"가 아니라, "AI와 함께 설계하고, AI가 실행하고, AI가 검증한다"는 시스템입니다.

---

## 이 문서를 읽기 전에

이 프로젝트(investmate)는 **Claude Code**라는 AI 코딩 도구로 개발됩니다. Claude Code는 단순한 챗봇이 아니라, 파일을 읽고 쓰고, 명령어를 실행하고, 테스트를 돌리는 **자율 개발 에이전트**입니다.

마치 회사에서 **시니어 개발자가 주니어 팀원 10명을 지휘하는 것**과 비슷합니다. 사용자가 "이거 만들어줘"라고 하면, Claude Code가 계획을 세우고, 전문 팀원(에이전트)에게 일을 나눠주고, 결과를 검증합니다.

## 2. 핵심 개념 4가지

### 2-1. CLAUDE.md — "프로젝트 설명서"

프로젝트 루트에 있는 `CLAUDE.md` 파일입니다. Claude Code는 **매 대화 시작 시** 이 파일을 읽습니다.

**비유:** 새 직원이 입사하면 받는 "업무 매뉴얼"

이 파일에 적혀 있는 것:
- 이 프로젝트가 뭐하는 건지 (S&P 500 종목 추천)
- 기술 스택이 뭔지 (Python, FastAPI, SQLite)
- 파이프라인이 어떻게 돌아가는지 (STEP 0~6)
- DB 구조, 설정값, CLI 명령어 등

**왜 중요한가:** Claude Code가 "AAPL이 뭔지", "return_20d가 뭔 컬럼인지" 물어보지 않고 바로 코딩할 수 있는 이유가 이 파일 덕분입니다.

### 2-2. Rules — "일하는 규칙"

`~/.claude/rules/` 폴더에 있는 규칙 파일들입니다.

**비유:** 회사의 "코딩 컨벤션 가이드"

```
rules/
├── common/           ← 모든 프로젝트에 적용
│   ├── coding-style.md    "변수를 직접 바꾸지 말고 새로 만들어"
│   ├── testing.md         "테스트를 먼저 쓰고 코드를 작성해"
│   ├── security.md        "비밀번호를 코드에 넣지 마"
│   ├── agents.md          "어떤 상황에 어떤 에이전트를 써"
│   └── ...
└── python/           ← Python 프로젝트에만 추가 적용
    ├── coding-style.md    "PEP 8 따르고, black으로 포맷팅해"
    └── ...
```

**핵심 규칙 예시:**

| 규칙 | 의미 |
|------|------|
| "불변성 준수" | 기존 객체를 수정하지 말고 새 객체를 만들어라 |
| "TDD 필수" | 테스트를 먼저 쓰고, 그 다음 코드를 작성해라 |
| "커버리지 80%" | 코드의 80% 이상이 테스트로 검증되어야 한다 |
| "코드 리뷰 필수" | 코드를 쓴 직후 반드시 code-reviewer 에이전트를 실행해라 |

### 2-3. Memory — "AI의 기억"

`~/.claude/projects/{프로젝트}/memory/` 폴더에 저장됩니다.

**비유:** 단골 카페에서 바리스타가 "아 이 손님 아이스 아메리카노 좋아하지"라고 기억하는 것

현재 저장된 기억들:

| 파일 | 내용 |
|------|------|
| `user_role.md` | "이 사용자는 30년차 퀀트 트레이더" → 전문 용어 사용 |
| `feedback_autonomous.md` | "중간에 물어보지 말고 알아서 진행해" |
| `feedback_workdir_restriction.md` | "이 폴더 밖의 파일은 건드리지 마" |

**작동 방식:**
1. 대화 중 사용자가 "나 퀀트 트레이더야"라고 말하면 → `user_role.md`에 저장
2. 다음 대화 시작 시 이 파일을 읽고 → "아, Sharpe ratio나 MDD 같은 전문 용어를 써야겠구나"
3. 사용자가 "자꾸 물어보지 마"라고 하면 → `feedback_autonomous.md`에 저장
4. 이후 대화에서 → 확인 질문 없이 바로 실행

### 2-4. Agents — "전문 팀원들"

Claude Code는 혼자 모든 걸 하지 않습니다. 복잡한 작업은 **전문 에이전트**에게 위임합니다.

**비유:** 건설 현장의 감독이 전기 기사, 배관 기사, 인테리어 업체에게 각각 일을 맡기는 것

```
~/.claude/agents/ (18명의 전문가)

주요 멤버:
├── planner.md         → 설계사: "이 기능을 어떻게 만들지 계획서를 써줘"
├── code-reviewer.md   → 품질 검사관: "이 코드에 문제 없는지 검토해줘"
├── tdd-guide.md       → 테스트 전문가: "테스트부터 작성하고 구현해줘"
├── architect.md       → 건축가: "시스템 구조를 어떻게 잡을지 설계해줘"
├── security-reviewer.md → 보안 전문가: "보안 취약점 있는지 확인해줘"
└── build-error-resolver.md → 소방관: "빌드가 깨졌으면 고쳐줘"
```

**에이전트가 자동으로 호출되는 상황:**

| 상황 | 자동 호출되는 에이전트 |
|------|---------------------|
| "이 기능 만들어줘" | planner → tdd-guide → code-reviewer |
| 코드를 수정한 직후 | code-reviewer (필수) |
| 빌드가 깨졌을 때 | build-error-resolver |
| 아키텍처 결정이 필요할 때 | architect |

**병렬 실행:**
독립적인 조사는 동시에 여러 에이전트를 실행할 수 있습니다.

```
예: "수익률 트래킹을 개선해줘"라고 요청하면

    동시에 3개 조사 시작:
    ├── 에이전트 1: 추천 종목 수 관련 코드 조사
    ├── 에이전트 2: 수익률 계산 코드 조사
    └── 에이전트 3: 신뢰성 관련 코드 조사
    
    → 3개가 동시에 돌아가서 시간 절약
```

## 3. 실제 개발 흐름 (방금 한 작업 기준)

### Step 1: 요청

```
나: "수익률 트래킹 시스템을 보완해야해"
```

### Step 2: 조사 (Explore Agent 3개 병렬)

Claude Code가 프로젝트를 이해하기 위해 3명의 조사원을 동시에 보냅니다.

```
조사원 1: "추천 종목 수가 어떻게 결정되는지 찾아봐"
    → config.py, feedback.py, screener.py 분석
    
조사원 2: "수익률 계산이 어떻게 되는지 찾아봐"
    → performance.py, api.py, templates 분석
    
조사원 3: "신뢰성 시스템이 뭐가 있는지 찾아봐"
    → calibrator.py, feedback.py, counterfactual.py 분석
```

3명이 동시에 작업해서 몇 분 만에 **프로젝트 전체 파악** 완료.

### Step 3: 설계 (Plan Agent)

조사 결과를 종합해서 설계 전문가에게 넘깁니다.

```
설계 전문가에게:
"현재 상태는 이렇고, 부족한 점은 이거야.
 Phase 1~4로 나눠서 구현 계획을 세워줘."

→ 상세한 계획서 반환 (어떤 파일을, 어떻게, 어떤 순서로)
```

### Step 4: 계획 모드 (Plan Mode)

계획서를 `.claude/plans/` 폴더에 저장하고, 사용자에게 보여줍니다.

```
Claude Code: "이렇게 계획했습니다. 승인하시겠습니까?"
나: (승인)
```

### Step 5: 구현

승인 후 코드 작성:

```
1. risk_metrics.py 생성 (리스크 지표 계산 함수)
2. performance.py 확장 (가중 수익률, 벤치마크, Sharpe 추가)
3. api.py 확장 (5개 신규 API)
4. performance.html 재설계 (대시보드 UI)
5. models.py 확장 (감사 추적 컬럼)
6. feedback.py 수정 (신뢰 구간)
7. calibrator.py 수정 (샘플 가드)
```

### Step 6: 검증

```
pytest tests/ -x -q
→ 1120개 테스트 전부 통과 ✓
```

### Step 7: 배포

```
git commit → git push → SSH로 EC2 접속 → 코드 받기 → 서비스 재시작
```

## 4. Skills (슬래시 명령)

Claude Code에서 `/명령어`로 특정 기능을 호출할 수 있습니다.

| 명령어 | 하는 일 | 비유 |
|--------|--------|------|
| `/plan` | 구현 계획 수립 | 설계도 그리기 |
| `/tdd` | 테스트 먼저 작성 후 구현 | 시험 문제 먼저 보고 공부 |
| `/code-review` | 작성한 코드 검토 | 선배한테 코드 봐달라고 하기 |
| `/commit` | Git 커밋 생성 | 작업 저장하기 |
| `/build-fix` | 빌드 에러 수정 | 고장난 거 고치기 |
| `/verify` | 전체 검증 (빌드+테스트+린트) | 출시 전 최종 점검 |

## 5. 자동 개선 루프

사람이 안 붙어있어도 Claude Code가 스스로 코드를 개선하는 시스템입니다.

```bash
make improve          # AI가 3번 반복하면서 코드 개선
make improve-coverage # 테스트 커버리지를 올려라
make improve-quality  # 코드 품질을 올려라
```

**작동 방식:**

```
반복 1:
  Claude Code가 코드를 분석하고 개선 → 테스트 실행
  ├── 테스트 통과 → 커밋하고 다음 반복
  └── 테스트 실패 → 변경 취소하고 다시 시도

반복 2: (같은 과정)
반복 3: (같은 과정)
```

한 번에 최대 25턴(=25번의 도구 사용)까지 작업하며, 테스트가 깨지면 자동으로 되돌립니다.

## 6. 일상적인 개발 명령어

### 터미널에서 쓰는 명령어

```bash
# Claude Code 시작
claude

# 코드 안에서 쓰는 명령어 (Makefile)
make test          # 테스트 실행
make coverage      # 커버리지 확인
make lint          # 코드 스타일 검사
make run           # 파이프라인 실행
make improve       # AI 자율 개선
```

### Claude Code 안에서 하는 대화

```
"이 기능을 만들어줘"          → planner + tdd-guide + code-reviewer 자동 호출
"이 버그를 고쳐줘"            → 코드 분석 → 수정 → 테스트
"배포해줘"                   → git push → EC2 SSH 배포
"이거 어떻게 돌아가는지 설명해줘" → Explore agent가 코드베이스 탐색
```

## 7. 왜 이런 시스템을 쓰는가?

### 전통적 개발
```
1. 사람이 코드를 쓴다
2. 사람이 테스트를 쓴다
3. 사람이 리뷰를 한다
4. 사람이 배포를 한다
→ 모든 단계에 사람이 필요
```

### Claude Code 개발
```
1. 사람이 "뭘 만들지" 설명한다
2. AI가 계획을 세운다 (planner agent)
3. AI가 코드를 쓴다 (tdd-guide agent)
4. AI가 리뷰를 한다 (code-reviewer agent)
5. AI가 테스트를 돌린다 (pytest)
6. 사람이 확인하고 배포를 승인한다
→ 사람은 "의사결정"에 집중, 반복 작업은 AI
```

### 메모리 시스템의 가치
```
일반 AI: 매번 "나 퀀트 트레이더야, 전문 용어 써줘" 설명해야 함
Claude Code: 한번 말하면 기억하고, 다음부터 자동 적용
```

## 8. 요약 — 한 문장으로
